# 依頼要約の表示に関する実装方針

## 背景
- 既存UIでは「依頼一覧」「ステータス一覧」に `order.summary` を表示している。
- 既存バックエンドでは会話ログ保存後に `OrderService.update_summary` で要約を更新している。
- ただし「新しい依頼を受け付けた直後」に一覧へ確実・即時反映するタイミングが明確でない。

## 目的
- 新規依頼受付後、生成した要約が「依頼一覧」「ステータス一覧」に速やかに反映される。
- フロントエンド/バックエンドそれぞれで反映タイミングを明確化し、データ不整合を抑える。

## 実装方針（全体）
1. **要約の正本はバックエンド（Order）に持つ**。
2. **要約更新のトリガーはバックエンドで統一**（会話ログ保存と同時）。
3. **フロントエンドは送信完了後に一覧を再取得**して、サーバー正本を反映する。
4. 必要に応じて段階的に「楽観更新（optimistic UI）」を追加する。

---

## フロントエンド方針（APIリクエストタイミング）

### A. 初期表示
- 画面マウント時に一覧取得（現状どおり `fetchOrders()`）。

### B. 新規依頼送信時
- 送信開始時: チャット領域のみ更新（ユーザー発話追加、進捗表示）。
- 送信完了時（SSE `done` 受信、または通常レスポンス成功）:
  - `session_id` 確定後に `fetchOrders()` を実行。
  - 一覧再取得後、該当オーダーをアクティブ表示（`session_id` で特定）。
- 送信失敗時:
  - 一覧再取得は行わない（誤反映を防止）。

### C. SSE利用時の注意
- `done` イベントを**唯一の再取得トリガー**にする。
- `phase` イベント中は一覧更新しない（要約未確定タイミングのため）。

### D. 追加改善（任意）
- 体感速度改善のため、`done` 前に暫定要約（先頭N文字）を表示してもよい。
- ただし最終的には必ず `fetchOrders()` の値で上書きする。

---

## バックエンド方針（データ登録タイミング）

### A. 新規依頼（NEW）
- `save_session` 実行後、`create_if_not_exists` で Order を確保。
- `conversation` に user/assistant を保存。
- 同トランザクション境界（または同一処理フロー）で `update_summary` を実行。
- これにより、一覧取得APIが返す `summary` は常に最新となる。

### B. 変更依頼（CHANGE）/照会（QUERY_STATUS）
- 現行同様、assistant応答が確定した時点で会話ログ保存 → `update_summary`。
- 「要約対象は最新ターン」か「初回依頼ベース」かは運用要件次第で切替可能。

### C. 失敗時の扱い
- 要約生成（LLM）が失敗してもフォールバック文字列を保存する。
- 会話ログ保存成功後に要約失敗した場合は、再試行可能な非同期ジョブ化を将来検討。

### D. 将来の永続化対応
- DB導入時は `orders` と `messages` の保存整合性を担保（同一トランザクション推奨）。
- `updated_at` を summary 更新時にも更新し、一覧ソート一貫性を維持。

---

## API仕様の見直し方針

### 現行のまま進める最小対応
- 既存 `POST /api/create_entry( /stream )` と `GET /api/orders` を継続利用。
- フロントで「完了後再取得」を徹底するだけで目的達成可能。

### 改善案（必要時）
- `create_entry` のレスポンスへ `order_id` / `summary` を含める。
  - メリット: 一覧再取得前でも該当行を特定しやすい。
  - デメリット: レスポンス責務が増える。

---

## 実装ステップ案
1. フロント: 送信成功後の `fetchOrders()` 呼び出し箇所を一本化（SSE done / fallback成功時）。
2. フロント: 再取得後に `session_id` で対象オーダーを選択する処理を追加。
3. バックエンド: NEW/CHANGE/QUERY_STATUS で `update_summary` が確実に呼ばれることをテストで固定。
4. テスト追加:
   - API: 依頼送信後に `GET /orders` の summary が更新されること。
   - FE（可能なら）: 送信完了時に orders 再取得が1回発火すること。

## 受け入れ基準
- 新規依頼送信完了後、画面遷移なしで依頼一覧・ステータス一覧に要約が反映される。
- 送信失敗時は一覧が誤って更新されない。
- 同一依頼の追加入力時、要約が最新会話に追随して更新される。
